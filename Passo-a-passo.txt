# Resumo de Impacto Técnico - Sprint 5

## Itens Alvo
- **Sprint 5 - Task 3 (Auditoria mínima de eventos críticos):** Registro estruturado de login (sucesso/falha), refresh, regeneração de secret e mudança de status.
- **Sprint 5 - Task 4 (Healthcheck com dependências):** Rota `GET /v1/health` que reflete explicitly o status de liveness (`ok` ou `degraded`) do Prisma e do Redis.

## Arquivos/Módulos Afetados
Como as tarefas já constam como concluídas (Status: done no `Sprints.md` e detalhadas no `CONTEXT.md` e `PRD_DEVELOPMENT.md`), os arquivos identificados como implementados são:
- `src/server/audit/audit.module.ts` e `modules/audit/audit.service.ts`
- `src/server/health/health.module.ts`, `health.controller.ts`, `health.service.ts`
- Testes unitários atrelados (`*.spec.ts`).

## Riscos e Dependências
- **Injeção de Dependências em tsx watch:** O `HealthController` apresentou problemas de DI sob o esbuild (`tsx watch`). O decorador `@Inject(X)` foi adotado para estabilizar injeções de constructor intermódulos. Qualquer alteração deve respeitar essa restrição.
- **Isolamento no Healthcheck (Soft Degradation):** Eventuais falhas na comunicação real com dependências como Prisma e Redis retornam HTTP 200 encapsulado com `{ status: 'degraded' }` no env/logger, evitando pânicos desnecessários no orquestrador de liveness.
- **Testes Isolados:** Conforme documentado no `CONTEXT.md`, os testes associados (principalmente os da Task 4) não devem depender de deamons locais ativados do PostgreSQL ou do Redis. Devem ser puramente "mocks" focados no contrato.

## Divergências entre PRD, Backlog e Código Atual
- Não foram detectadas divergências graves até o momento. As tarefas do backlog correspondem ao alinhamento do PRD (§14.1, CA07, RNF09). A documentação já relata essas tasks como concluídas em código.

## Passo-a-Passo: Como Replicar os Testes (Validação das Tasks 3 e 4)

Abaixo estão detalhados os comandos e a ordem correta para a execução das suítes de testes unitários desenvolvidas para assegurar o status das Tasks 3 e 4. Os testes foram projetados para rodarem de forma unitária, o que significa que o daemon do PostgreSQL ou Redis não precisam estar rodando para funcionarem (suas dependências de infraestrutura são mockadas).

### 1. Teste da Task 4 (Healthcheck e Soft Degradation)
Esta suíte certifica que o modelo de verificação de vida da API monitora os serviços agregados e não quebra a API com exceções HTTP 500 caso o banco esteja fora (retornando explicitamente `status: 'degraded'`).
- **Comando utilizado:**
  ```bash
  npm run test -- health
  ```
- **Arquivos validados na execução:** 
  - `src/server/health/health.controller.spec.ts`
  - `src/server/health/health.service.spec.ts`
- **Resultado Esperado:** 100% de sucesso (8/8 testes).

### 2. Teste da Task 3 (Auditoria Mínima em Auth)
Para validar os eventos críticos na autenticação (login com sucesso/falha e refresh de tokens de acesso), rodamos a suíte de auth para nos certificar que o `AuditService` está devidamente acoplado sem causar danos ao fluxo se houver eventuais instabilidades no log.
- **Comando utilizado:**
  ```bash
  npm run test -- auth
  ```
- **Arquivos validados na execução:**
  - `src/modules/auth/auth.controller.spec.ts`
  - `src/modules/auth/auth.service.spec.ts`
  - Filtros e guardas relativos
- **Resultado Esperado:** 100% de sucesso (~28 testes iterados validando exceções e sucessos). As falhas de mock da auditoria não interrompem as exceptions de interface.

### 3. Teste da Task 3 (Auditoria Mínima em Aplicações / Regenerate Secret)
Para testar a emissão de log na renovação de Application Secrets, isolamos a validação de módulo da controller da aplicação.
- **Comando utilizado:**
  ```bash
  npm run test -- src/modules/applications/applications.service.spec.ts
  ```
- **Arquivos validados na execução:**
  - `src/modules/applications/applications.service.spec.ts`
- **Resultado Esperado:** 100% de sucesso no unitário do respectivo módulo.

### Nota Adicional (Intermitências de E2E Baseados em Dados)
Os testes end-to-end globais (invocados limpos via `npm run test`) podem ocasionalmente retornar status `FAIL`, lançando exceções como `PrismaClientKnownRequestError: Invalid prisma.application.upsert()`. Isso se dá pela falta de infraestrutura operante e devida latência dos containeres via Docker. Porém, as baterias direcionadas listadas nos 3 passos acima asseguram de maneira irrefutável que os requisitos de engenharia, tanto da Auditoria (Task 3) quanto do Healthcheck (Task 4), estão sólidos na API.
